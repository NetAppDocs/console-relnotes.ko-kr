=== Cloud Volumes ONTAP



==== AWS의 Cloud Volumes ONTAP 및 데이터 서비스에 대한 유연한 요금 청구

NetApp은 AWS에서 Cloud Volumes ONTAP 구독에 대한 향상된 청구 옵션을 제공하여 Cloud Volumes ONTAP 용량 및 BlueXP 백업 및 복구와 같은 NetApp 데이터 서비스 비용 지불 방식을 더욱 세밀하게 제어할 수 있도록 지원합니다. 이전에는 모든 Cloud Volumes ONTAP 및 데이터 서비스 사용량이 AWS 마켓플레이스의 NetApp Intelligent Services(NIS) 목록을 통해 통합적으로 청구되었습니다.

AWS 엔터프라이즈 할인 프로그램(EDP) 변경으로 인해 NetApp은 AWS 마켓플레이스에 독립형 Cloud Volumes ONTAP 구독을 출시했습니다. 이를 통해 Cloud Volumes ONTAP 용량과 데이터 서비스 요금을 분리하여 청구할 수 있습니다. 이러한 분리를 통해 EDP 크레딧 및 비용 관리에 있어 더욱 유연성을 확보할 수 있습니다. NetApp은 Cloud Volumes ONTAP 사용량에 대한 비용 관리를 개선하기 위해 독립형 구독을 사용하는 것을 권장하며, EDP 요건이 적용되는 2026년 7월 31일 이후에는 데이터 서비스 요금 청구를 위해 NIS 구독을 계속 사용할 것을 권장합니다.

AWS의 독립형 Cloud Volumes ONTAP 구독은 유효한 용량 기반 Essentials 또는 Freemium 라이센스 패키지만 지원합니다. Professional 패키지는 지원되지 않습니다. 필요에 가장 적합한 청구 방법을 선택할 수 있습니다.

* *독립형 Cloud Volumes ONTAP 구독(권장)*: AWS 마켓플레이스에서 독립형 구독을 통해 Cloud Volumes ONTAP 사용량에 대한 비용을 지불하고, 데이터 서비스 요금은 NIS 구독을 통해 계속 청구됩니다. 이 옵션은 NetApp Console의 AWS 계정에 독립형 및 NIS 구독이 모두 연결된 대부분의 고객에게 권장됩니다.
* *통합 NIS 구독*: 기존 NIS 목록을 통해 Cloud Volumes ONTAP과 데이터 서비스를 함께 결제하세요. 이 통합 방식은 NetApp Console에서 AWS 계정에 NIS 구독만 연결된 기존 고객에게 적용됩니다. 하지만 NetApp은 더 나은 비용 관리를 위해 Cloud Volumes ONTAP 사용량에 대한 결제 기본 설정을 지정하고 독립형 구독으로 전환하는 것을 권장합니다.


구독 유형에 따라 다음과 같은 시나리오가 제공됩니다.

.독립형 구독만 가능(신규 고객)
NetApp Console의 AWS 계정에 독립형 AWS 구독만 연결되어 있고 NIS 구독이 없는 경우, 배포 시 Essentials 라이선스 패키지만 선택할 수 있습니다. 데이터 서비스 *Backup and Recovery* 옵션을 사용할 수 없습니다.

.독립 실행형 및 NIS 구독(기존 고객) 모두
AWS 계정에 NIS 구독과 독립 실행형 구독이 모두 연결되어 있고 NetApp Console의 *청구 기본 설정* 탭에서 청구 기본 설정을 지정하지 않은 경우 Cloud Volumes ONTAP 사용량은 기본적으로 NIS를 통해 청구됩니다. 독립 실행형 목록에서는 Professional을 지원하지 않으므로 Professional 패키지를 실행하는 모든 Cloud Volumes ONTAP 인스턴스에서도 기본적으로 이 방식이 적용됩니다. 청구 기본 설정을 지정하여 각 서비스에 대한 청구 방식을 선택할 수 있습니다.

* 독립형 구독을 Cloud Volumes ONTAP(Essentials 패키지 포함)의 기본 결제 계정으로 설정하세요. Professional 패키지만 있는 경우 Essentials 패키지로 전환하고 결제 기본 설정을 지정하여 Cloud Volumes ONTAP 사용량을 청구할 수 있습니다. 이 방법은 Cloud Volumes ONTAP EDP 크레딧을 보다 효과적으로 관리하는 데 도움이 되므로 권장합니다.
* NIS 구독을 사용하여 데이터 서비스 요금을 청구하거나 Professional 패키지를 통해 Cloud Volumes ONTAP 사용량을 청구합니다. 배포 중에 *Backup and Recovery* 데이터 서비스를 활성화하고 Professional 라이센스 패키지를 선택하여 NIS 구독을 통해 데이터 서비스 요금을 청구합니다.


.NIS 구독 전용(기존 고객)
이는 AWS 계정과 연결된 NIS 구독을 보유한 기존 고객에게 적용됩니다. 현재와 같이 Cloud Volumes ONTAP과 데이터 서비스를 함께 청구할 수 있습니다. 하지만 NetApp은 Cloud Volumes ONTAP 사용량에 대한 비용 관리를 개선하기 위해 독립형 구독 방식으로 전환하는 것을 권장합니다.

* AWS 포털에서 새로운 독립형 목록을 구독하세요.
* 독립형 구독을 AWS 계정에 연결하세요.
* Essentials 패키지를 사용한 Cloud Volumes ONTAP 사용에 대해 결제 기본 설정을 독립형 구독으로 설정하십시오.
* (해당되는 경우) Professional 패키지를 Essentials 패키지로 변경하고 결제 방식을 설정하세요.
* NIS 구독을 통해 데이터 서비스 요금 청구를 계속 진행하십시오.



NOTE: 여러 구독 유형을 사용하거나 목록 간에 마이그레이션하는 경우 중복 측정이 발생하지 않습니다.

==== Hyper-V 워크로드 개선 사항

이번 릴리스에서 Hyper-V 워크로드는 다음과 같은 개선 사항을 도입했습니다.

* *SAN 스토리지의 VM 보호*: 이제 SAN 객체 스토리지에 있는 VM을 보호할 수 있습니다.
* *대체 위치로 복원 기능 향상*: 이제 SAN 스토리지에 있는 VM을 기본 및 보조 위치에서 대체 위치로 복원할 수 있습니다.
* *게스트 VM으로 파일 및 폴더 복원*: 이제 기본 스토리지의 스냅샷에서 게스트 Windows VM으로 파일과 폴더를 복원할 수 있습니다. 이 기능은 CIFS VM에서만 지원됩니다.


Hyper-V 워크로드 보호에 대한 자세한 내용은 https://docs.netapp.com/us-en/data-services-backup-recovery/br-use-hyperv-protect-overview.html["Hyper-V 워크로드 보호 개요"]를 참조하십시오.

=== 재해 복구



==== Kubernetes 기반 워크로드 지원(미리 보기)

NetApp Disaster Recovery에 Kubernetes 기반 워크로드 지원이 추가되었습니다. Kubernetes 지원을 통해 OpenShift Virtualization 환경을 포함한 Kubernetes 클러스터에서 실행되는 애플리케이션을 보호할 수 있습니다. Disaster Recovery는 link:https://docs.netapp.com/us-en/trident/trident-protect/trident-protect-requirements.html["Trident Protect에서 현재 지원하는 모든 Kubernetes 버전"^]를 지원하여 컨테이너화된 애플리케이션에 대한 원활한 통합과 강력한 재해 복구 기능을 제공합니다.

Kubernetes 기반 워크로드 지원의 주요 기능:

* **폭넓은 호환성**: Disaster Recovery는 Red Hat OpenShift Virtualization을 포함하여 Trident Protect와 호환되는 모든 Kubernetes 버전을 지원합니다.
* **애플리케이션 중심 보호**: 이제 애플리케이션 수준에서 Kubernetes 워크로드를 보호하여 페일오버 이벤트 발생 시 중요한 서비스의 복구를 보장할 수 있습니다.
* **간소화된 오케스트레이션**: Disaster Recovery는 VMware 환경에서 사용되는 것과 동일한 효율적인 SnapMirror 기술을 활용하여 Kubernetes 워크로드의 백업 및 복제를 자동화하는 직관적인 워크플로우를 제공합니다.


Kubernetes 기반 워크로드 지원은 현재 미리 보기로 제공됩니다. Kubernetes 기반 워크로드 보호를 시작하려면 link:https://docs.netapp.com/us-en/data-services-disaster-recovery/get-started/task-kubernetes-requirements.html["NetApp Disaster Recovery를 위한 Kubernetes 요구 사항"^]을(를) 참조하십시오.



=== 랜섬웨어 복원력



==== 클린 복원 기능 향상

이제 클린 복원 기능을 통해 다른 위치로 복구할 수 있습니다. 이를 통해 동일한 Console 에이전트에서 관리하는 다른 시스템, 스토리지 VM 및 애그리게이트로 볼륨을 복원할 수 있습니다. 또한, 클린 복원 기능은 복구에 사용할 수 있는 스냅샷을 검색하는 데 적용되던 7일 제한을 해제했습니다. 이제 클린 복원 기능은 랜섬웨어 발생 시점과 관계없이 사용 가능한 스냅샷을 지속적으로 검색하여 복구 지점을 찾습니다.

link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/task-clean-restore-recover.html["깔끔한 복원을 통해 워크로드 복구"^].



==== Google SecOps SIEM 지원

Ransomware Resilience는 이제 Google SecOps를 통한 보안 정보 및 이벤트 관리(SIEM)를 지원하므로 간소화된 위협 분석 및 탐지를 위해 데이터를 Google SecOps로 자동으로 전송할 수 있습니다.

구성 정보는 link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/task-siem.html["Ransomware Resilience를 SIEM에 연결하세요"^]을(를) 참조하십시오.



==== Azure NetApp Files 지원이 이제 정식으로 제공됩니다

Ransomware Resilience는 이제 Azure NetApp Files 시스템을 지원하여 Azure NetApp Files에서 랜섬웨어 위협을 효율적으로 탐지하고 대응할 수 있습니다. 워크로드를 검색할 때 Ransomware Resilience는 이제 Azure NetApp Files를 표시하고 보호 대시보드에 표시합니다. Azure NetApp Files에 대한 Ransomware Resilience 지원에는 스냅샷만을 사용한 탐지 및 보호 전략이 포함됩니다. Azure NetApp Files 지원은 현재 정식 출시(GA)되었습니다.

자세한 내용은 link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/concept-ransomware-resilience.html["랜섬웨어 복원력에 대해 알아보세요"^]를 참조하십시오.



==== SOAR의 사용자 차단 지원

Ransomware Resilience이 이제 Google SecOps 및 Splunk SOAR 플레이북을 사용하여 개별 사용자 차단을 지원합니다. 이 추가 기능을 통해 Ransomware Resilience에서 데이터 유출이나 대량 데이터 삭제와 같은 의심스러운 사용자 활동을 감지할 때 사용자 차단을 자동화할 수 있습니다.

link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/reference-soar.html["SOAR 플레이북 통합"^].

기존 고객은 구독 또는 라이선스 계약이 만료될 때까지 NetApp Cloud Tiering을 계속 사용하고 지원을 받을 수 있습니다. 구독이 만료된 후에는 고객이 NetApp Cloud Tiering 기능이나 지원에 더 이상 액세스할 수 없습니다.

=== 재해 복구



==== Google Cloud VMware Engine(GCVE) 지원이 이제 정식 출시(GA)되었습니다.

NetApp Disaster Recovery의 GCVE 지원이 이제 일반 공급(GA)으로 제공됩니다. GA 지원을 통해 GCVE용 Disaster Recovery는 이제 다음을 지원합니다:

* *플랫폼 관리형 스냅샷:* ONTAP 또는 Google Cloud Platform(GCP)을 통해 백업 및 보존 정책을 원활하게 관리합니다.
* *복제 계획당 여러 데이터 저장소:* 여러 데이터 저장소를 단일 복제 전략으로 통합하여 재해 복구 계획을 간소화하십시오.
* *사용할 수 없는 소스 사이트에 대한 강제 페일오버:* 소스를 사용할 수 없는 경우 재해 복구 사이트로 전환합니다.
* *SnapMirror 재동기화 연기 및 필요에 따라 복제 복원:* "보호 건너뛰기" 옵션을 사용하여 운영량이 많은 시간대에 프로덕션 워크로드에 미치는 영향을 방지하기 위해 SnapMirror 재동기화를 연기하고, 나중에 재보호 옵션을 사용하여 필요에 따라 복제를 복원할 수 있습니다


GCVE 지원에 대한 자세한 내용은 link:https://docs.netapp.com/us-en/data-services-disaster-recovery/release-notes/dr-limitations.html["Disaster Recovery를 위한 지원 구성"^]을(를) 참조하십시오.



=== 랜섬웨어 복원력



==== 클린 복원

Ransomware Resilience는 이제 랜섬웨어 공격으로부터 빠르고 쉽게 복구할 수 있도록 지원하는 새로운 복구 옵션인 클린 복원을 지원합니다. 클린 복원은 다운타임을 최소화하면서 워크로드를 가동하고 실행하기 위한 최적의 복원 지점을 식별하는 안내형 워크플로를 제공합니다. 클린 복원의 워크플로는 또한 데이터에 대한 파일 수준의 인사이트를 제공하여 재감염을 방지하고 데이터 손실을 최소화하는 멀웨어 없는 워크로드를 복원할 수 있도록 보장합니다.

지원되는 구성 및 클린 복원을 위한 시스템 구성 방법에 대한 자세한 내용은 link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/task-clean-restore.html["격리된 복구 환경 생성"^]를 참조하십시오. 클린 복원을 사용하여 복구하는 방법에 대한 자세한 내용은 link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/task-clean-restore-recover.html["깔끔한 복원을 통해 워크로드 복구"^]를 참조하십시오.



==== Splunk Enterprise용 SIEM 지원

Ransomware Resilience는 이제 Splunk Enterprise용 보안 정보 이벤트 관리(SIEM) 지원을 제공하여 로그 및 이벤트 데이터를 중앙 집중화하고 위협 분석 및 탐지를 간소화할 수 있습니다.

자세한 내용은 link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/task-siem.html["Ransomware Resilience를 SIEM에 연결하세요"^]를 참조하십시오.

=== Cloud Volumes ONTAP



==== Google Cloud C3 VM에서 Cloud Volumes ONTAP 지원이 향상되었습니다.

Cloud Volumes ONTAP는 이제 Google Cloud C3 시리즈 VM에 대한 지원을 강화하여 다음과 같은 추가 기능을 사용할 수 있게 되었습니다.

* Cloud Volumes ONTAP 배포에서 C3 제품군 내의 다양한 변형 간에 인스턴스 유형을 전환할 수 있습니다.
* 인스턴스 유형을 변경할 때 대상 VM으로 c3-standard-44 또는 c3-standard-88 및 해당 LSSD 변형을 선택하면 Google 네트워크 서비스 송신 네트워크 계층 Tier-1이 자동으로 적용됩니다. 다른 C3 변형으로 전환하면 송신 네트워크 계층은 Standard로 설정됩니다.
+

NOTE: 송신 네트워크 계층을 Tier-1으로 변경하면 추가 비용이 발생합니다. 배포 및 업그레이드 계획을 적절히 수립하십시오.

* C3의 로컬 SSD(LSSD) 변형을 선택하면 Flash Cache가 기본적으로 활성화됩니다. 대상 변형이 Flash Cache를 지원하더라도 Cloud Volumes ONTAP 배포 환경에서는 C3 LSSD 변형 간에 전환할 수 없습니다.
* 이제 C3 VM에서 새 애그리게이트와 기존 애그리게이트 모두에 볼륨을 추가할 수 있습니다.
* NetApp Console을 통해 또는  `PUT /gcp/vsa/aggregates/{workingEnvironmentId}/{aggregateName}` API를 사용하여 볼륨을 추가할 때 IOPS 및 처리량 설정을 변경할 수 있습니다.

==== Cloud Volumes ONTAP에 대한 Console 에이전트 프록시 설정을 관리하는 기능

이제 연결이 끊어지거나 프록시 설정이 잘못된 경우에도 NetApp Console 에이전트를 사용하여 Cloud Volumes ONTAP의 프록시 설정을 관리할 수 있습니다. 이전에는 Console 에이전트가 20분 동안 Cloud Volumes ONTAP에 연결할 수 없는 경우 수동 프록시 설정을 기본 설정으로 덮어썼습니다. 이로 인해 AutoSupport 메시지 관련 문제를 포함한 통신 오류가 발생했습니다. 기존 시스템의 프록시 설정을 유지하려면 다음 API 호출을 실행하십시오.

`PUT /occm/config`

요청 본문에 다음 매개변수를 포함하십시오.

[source, json]
----
{
  "proxyMode": "No_Overwrites"
}
----
기본 모드는 표준 모드입니다. 즉, Console 에이전트가 20분 동안 Cloud Volumes ONTAP에 연결할 수 없는 경우 프록시 설정을 기본값으로 덮어씁니다.

https://docs.netapp.com/us-en/console-automation/cvo/put-occm-config.html["수정 가능한 NetApp Console 매개변수를 구성합니다"^]

=== Cloud Volumes ONTAP



==== 차세대 Google Cloud VM을 위한 Cloud Volumes ONTAP 지원

9.18.1에서 NetApp은 새로운 Cloud Volumes ONTAP 배포를 N2 VM에서 차세대 Google Cloud C3 시리즈 VM으로 전환하여 더욱 빠르고 확장 가능한 환경을 제공합니다. 이제 Google Cloud에서 Cloud Volumes ONTAP 9.18.1 이상 버전을 배포할 때 C3 시리즈 VM의 이점을 활용할 수 있습니다. C3 시리즈 머신은 Google Virtual NIC(gVNIC)와 Hyperdisk Balanced 디스크를 사용하여 향상된 성능과 더 높은 용량 제한을 제공하며, 집약적인 워크로드에 필요한 동적 성능을 보장합니다.


NOTE: 현재 Cloud Volumes ONTAP는 단일 노드 배포 환경에서만 C3 시리즈를 지원합니다.

Cloud Volumes ONTAP 시스템이 9.18.1 이상을 실행하는 경우 간편한 단일 노드 배포에 사용하는 사전 구성된 패키지는 자동으로 C3 VM을 사용하며 워크로드 요구 사항에 따라 IOPS 및 처리량 매개 변수를 사용자 지정할 수 있습니다. 마찬가지로 애그리게이트를 생성하는 동안 Hyperdisk Balanced 디스크를 추가하여 Google Cloud에서 더 나은 성능과 확장성을 달성할 수 있습니다. 또한 기본 Flash Cache 지원을 위해 C3 시리즈 머신의 LSSD 변형을 선택할 수 있습니다.

C3 VM은 Hyperdisk Balanced 디스크만 지원하기 때문에 애그리게이트에 볼륨을 추가할 때 디스크 유형을 변경할 수 없습니다. 마찬가지로 N2 VM 유형의 시스템을 C3 VM으로 복제할 때 디스크 유형은 기본적으로 Hyperdisk Balanced로 설정됩니다.

https://docs.netapp.com/us-en/cloud-volumes-ontap-relnotes/reference-configs-gcp.html["Google Cloud에서 Cloud Volumes ONTAP에 대해 지원되는 구성"^]

https://docs.cloud.google.com/compute/docs/general-purpose-machines#c3_series["Google 문서: C3 머신 시리즈"^]



==== Azure의 Cloud Volumes ONTAP용 VNet 보안

Azure 단일 및 다중 가용성 영역에 배포된 Cloud Volumes ONTAP 9.18.1 이상 버전은 전송 중인 데이터를 보호하기 위한 계층형 보안 전략의 일환으로 Azure 가상 네트워크(VNet) 암호화를 지원합니다. Cloud Volumes ONTAP는 Azure의 기본 데이터그램 전송 계층 보안(DTLS) 프로토콜을 활용하여 ONTAP 노드, 관리 인터페이스 및 기타 Azure 서비스 간의 통신을 보호하고, 데이터 가로채기 및 무단 액세스를 방지합니다. 이러한 네트워크 수준 암호화는 ONTAP에 내장된 스토리지 및 저장 데이터 보호 기능을 보완하여 데이터에 대한 엔드 투 엔드 보안을 제공합니다.

https://review.docs.netapp.com/us-en/storage-management-cloud-volumes-ontap_gcp-sn-c3-support/reference-networking-azure.html#networking-for-azure-vnet-encryption["Azure VNet 암호화를 위한 네트워킹"^]

=== 랜섬웨어 복원력



==== Azure NetApp Files 지원

Ransomware Resilience가 이제 Azure NetApp Files 시스템을 지원하여 Azure NetApp Files에서 랜섬웨어 위협을 효율적으로 탐지하고 대응할 수 있습니다. 워크로드를 검색할 때, Ransomware Resilience는 이제 Azure NetApp Files를 표시하고 이를 보호 대시보드에 보여줍니다. Azure NetApp Files에 대한 Ransomware Resilience 지원은 스냅샷만을 사용한 탐지 및 보호 전략을 포함합니다. Azure NetApp Files 지원은 현재 프리뷰 단계입니다.

자세한 내용은 link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/concept-ransomware-resilience.html["랜섬웨어 복원력에 대해 알아보세요"^]를 참조하십시오.



==== 사용자 행동 알림에서 사용자 제외

NetApp Ransomware Resilience를 사용하면 이제 사용자 행동 알림에서 특정 사용자를 제외할 수 있습니다. 신뢰할 수 있는 사용자를 제외하면 오탐 및 불필요한 알림을 방지할 수 있습니다.

자세한 내용은 link:For more information, see link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/suspicious-user-activity.html#exclude-users-from-alerts["알림에서 사용자 제외"^]을 참조하십시오.



==== 사용자 행동 활동에 대한 보호 그룹 지원

Ransomware Resilience 보호 그룹에서 이제 의심스러운 사용자 행동 탐지를 위한 탐지 정책을 지원합니다. 보호 그룹에 랜섬웨어 보호 전략을 적용하면 워크로드 전반에 정책이 적용되어 사이버 보안 태세 관리가 간소화됩니다.

자세한 내용은 link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/rp-use-protect.html#create-a-protection-group["보호 그룹 생성"^]를 참조하세요.



=== ONTAP 용 Amazon FSX



==== 스토리지 VM 마이그레이션 지원

NetApp Workload Factory에서 이제 스토리지 VM 마이그레이션을 지원합니다. 이 기능을 사용하면 온프레미스 ONTAP 시스템 또는 1세대 FSx for ONTAP 파일 시스템에서 2세대 FSx for ONTAP 파일 시스템으로 ONTAP 스토리지 시스템 데이터 및 구성을 마이그레이션할 수 있습니다. 스토리지 VM 데이터와 구성 설정을 복제하여 사용자 및 애플리케이션에 대한 다운타임과 중단을 최소화하면서 새 파일 시스템으로 이동할 수 있습니다.

이 기능을 사용하려면 link:https://docs.netapp.com/us-en/storage-management-fsx-ontap/use/task-replicate-data.html["복제 관계 생성"] 사용 사례로 *Migration*을 선택하십시오. 마이그레이션 프로세스를 완료하려면 link:https://docs.netapp.com/us-en/storage-management-fsx-ontap/use/task-replicate-data.html#cut-over-replication-for-migration-use-cases["스토리지 VM과 복제된 볼륨을 컷오버합니다"] 즉시 데이터와 스토리지 VM 구성 설정을 타겟 FSx for ONTAP 파일 시스템으로 영구적으로 마이그레이션해야 합니다.
